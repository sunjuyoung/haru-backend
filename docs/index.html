CrewAI가 Python 네이티브 프레임워크이기 때문입니다. Spring Boot로 백엔드를 구성하면 AI 파이프라인을 별도 Python 서버로 분리해야 하고, 서버 간 통신 오버헤드 + 배포 복잡도가 증가합니다. 하루일기는 AI 파이프라인이 핵심 비즈니스 로직 그 자체이므로, 동일 프로세스에서 에이전트를 호출하는 것이 지연시간과 운영 복잡도 측면에서 유리합니다.

트레이드오프: FastAPI는 Spring Boot 대비 엔터프라이즈 생태계(보안, 트랜잭션, AOP)가 얇지만, Phase 1 규모(DAU 100 미만)에서는 SQLAlchemy async + Pydantic 조합으로 충분합니다.

일기 생성은 단방향 데이터 흐름입니다 — 서버가 진행 상황을 push하고 클라이언트는 받기만 합니다. WebSocket의 양방향 채널은 과잉이며, SSE는 HTTP 위에서 동작하므로 CDN/프록시 친화적이고 FastAPI의 StreamingResponse로 자연스럽게 구현됩니다.

핵심 문제 — CrewAI 동기 실행 vs asyncio

CrewAI는 내부적으로 동기 함수로 에이전트를 실행합니다. 그러나 FastAPI의 SSE 스트리밍은 asyncio 이벤트 루프 위에서 동작합니다. 동기 에이전트를 직접 호출하면 이벤트 루프가 블로킹되어 SSE 프로그레스 전송이 멈춥니다.

services/pipeline.py — asyncio.wait_for + 글로벌 Executor 
AGENT_MAX_WORKERS = int(os.getenv("AGENT_MAX_WORKERS", "4"))
AGENT_TIMEOUT_SECONDS = int(os.getenv("AGENT_TIMEOUT_SECONDS", "30"))

# 글로벌 executor — 프로세스 전체에서 공유
executor = ThreadPoolExecutor(max_workers=AGENT_MAX_WORKERS)

#CrewAI의 동기 함수를 별도 스레드에서 실행하므로, asyncio 이벤트 루프는 블로킹되지 않습니다. 
#이벤트 루프가 자유로우니 SSE 스트리밍도 계속 동작합니다.
#추가로 시간 제한까지 걸어서, CrewAI가 30초 이상 응답하지 않으면 asyncio 레벨에서 취소합니다.
async def run_agent_with_timeout(func, *args, timeout=AGENT_TIMEOUT_SECONDS):

    loop = asyncio.get_event_loop()
    return await asyncio.wait_for(
        loop.run_in_executor(executor, partial(func, *args)),
        timeout=timeout,
    )

핵심/부가 에이전트 분리 — Exception 격리

가장 중요한 의사결정은 에이전트를 핵심(critical)과 부가(optional)로 분류하고, 실패 처리를 구조적으로 분리한 것입니다.

분류에이전트실패 시 동작근거
핵심Agent 2, 3예외 전파 → SSE error → 재시도 안내결과 화면 구성 불가
부가Agent 5memory = None → 카드 비노출핵심 결과만으로 완전한 경험

스레드 모델의 한계: asyncio.wait_for timeout이 발동해도, run_in_executor로 제출된 동기 HTTP 호출(GPT-4o, Replicate)은 스레드에서 자연 종료까지 점유됩니다. Phase 2에서 httpx 클라이언트 레벨 timeout 추가를 검토 중입니다.

max_workers 튜닝: max_workers=4는 동시 요청 1~2건 기준입니다. 요청당 3 task이므로, 동시 2요청이면 6스레드 필요 → executor 포화 가능. 모니터링 후 동적 조정 또는 큐잉 검토.

핵심 의사결정

브라우저의 EventSource API는 GET만 지원합니다. 일기 생성 요청은 POST body에 인증 토큰과 내용을 담아야 하므로 EventSource로는 구현할 수 없습니다.

추가 이점: EventSource의 자동 재연결은 이 유스케이스에서 오히려 위험합니다. 파이프라인 중간에 끊기면 처음부터 다시 실행해야 하는데, 자동 재연결은 중복 실행을 야기합니다.

A안: 에이전트 완료 기반

+ 정확한 진행 상황

− Replicate 10~30초 편차 → UX 불예측

B안: 시간 기반 ✓

+ 일정한 리듬감, 명상적 대기 경험

− 실제 진행과 불일치 가능

하루일기는 "사색적 경험"이 핵심 컨셉. 정확한 진행률보다 일정한 리듬으로 전환되는 메시지가 연필 소리 ASMR과 함께 명상적 대기 경험을 만듭니다.

[사용자] ── "오늘 일기 다시 쓸래요" ──→ [확인 다이얼로그]
    ├── R2 URL 수집 (삭제 안 함)
    ├── DB soft delete
    ├── SSE 파이프라인 시작
    │   ├─ 성공 → DB 저장 → R2 비동기 삭제
    │   └─ 실패 → DB rollback → R2 보존 → 완전 복구

핵심: "실패해도 사용자가 잃는 것이 없다"는 보장.

  • POST /api/interview/{id}/answer
  • Evaluator → 채점 (1~10점)
  • Interviewer → 꼬리질문 or 다음 질문
  • 메인 질문당 꼬리질문 최대 2회
→
Phase 3 — 리포트 생성
  • POST /api/interview/{id}/end
  • Coach 에이전트 실행
  • 종합 점수 + 등급 (A/B/C/D/F)
  • 강점/보완점 정리
  • 지식 갭 분석 (학습O/X 분류)
  • 학습 추천 + 다음 단계 제시

01

비동기 면접 준비 처리

Spring @Async React Query 세션 상태 머신
⚠ Problem
Phase 1 AI 호출(CrewAI 5개 에이전트 순차 실행)이 10~30초 소요되어, 동기 블로킹 시 사용자가 빈 화면에서 대기해야 하는 UX 문제 발생
✅ Solution
Spring @Async로 백그라운드 준비 처리 + 세션 상태(PREPARING → READY / FAILED) 관리. 프론트엔드에서 React Query refetchInterval로 5초 폴링하고, PREPARING 세션이 없으면 폴링 자동 중단

세션 상태 흐름

세션 생성 요청 → PREPARING → READY / FAILED

백엔드 — @Async 비동기 처리

@Async public void prepareInterviewAsync(InterviewSession session) { try { // CrewAI Phase 1: 10~30초 소요 ScenarioResult result = aiClient.createScenario(...); session.markReady(result.getScenario()); } catch (Exception e) { session.markFailed(e.getMessage()); } sessionRepository.save(session); }

프론트엔드 — React Query 조건부 폴링

const { data } = useQuery({ queryKey: ['sessions'], queryFn: fetchSessions, // PREPARING 세션이 있을 때만 5초 간격 폴링 refetchInterval: data?.some( s => s.status === 'PREPARING' ) ? 5000 : false, });
이 패턴으로 사용자는 면접 준비 중에도 다른 작업을 할 수 있고, 준비가 완료되면 자동으로 UI가 갱신됩니다. 불필요한 폴링도 PREPARING 세션이 없으면 즉시 중단됩니다.
02

CrewAI 멀티 에이전트 오케스트레이션

CrewAI 에이전트 격리 프롬프트 엔지니어링
⚠ Problem
5개 에이전트의 역할 분담, 태스크 간 데이터 전달, 매 턴 새 인스턴스 생성이 필요. 특히 하나의 에이전트에 여러 async 태스크를 할당하면 OpenAI Tool Calling의 순서 제약으로 인해 대화 맥락이 충돌하는 치명적 에러 발생
✅ Solution
Phase별 Crew 분리 + context 체이닝으로 태스크 간 결과 전달. Phase 2는 팩토리 패턴으로 매 턴 새 Crew 생성(CrewAI stateless 특성 활용). async 태스크는 에이전트를 분리하여 Tool Call 충돌을 근본적으로 해결

문제: 에이전트 공유 시 Tool Call 순서 충돌

OpenAI의 Tool Calling은 엄격한 요청-응답 순서를 요구합니다. 하나의 에이전트가 여러 async 태스크를 동시에 처리하면 Tool Call ID가 꼬여서 에러가 발생합니다.
Agent (Task A) → call_A 요청 → Task C가 call_C 끼어듦 → 순서 불일치 에러
// ✗ 에러 발생: 동일 에이전트에 async 태스크 3개 할당 analyst = create_analyst() task_a = Task(agent=analyst, async_execution=True) # 이력서 task_b = Task(agent=analyst, async_execution=True) # JD task_c = Task(agent=analyst, async_execution=True) # RAG # → "tool_calls must be followed by tool results" 에러

해결: 에이전트 인스턴스 분리

// ✓ 해결: 태스크별 별도 에이전트 인스턴스 resume_analyst = create_analyst(user_id) # 이력서 전담 jd_analyst = create_analyst(user_id) # JD 전담 rag_analyst = create_analyst(user_id) # RAG 전담 # 각 에이전트가 독립적인 대화 컨텍스트를 유지
각 에이전트가 독립적인 LLM 대화 세션을 가지므로, Tool Call ID 충돌 없이 3개 분석 태스크가 안전하게 병렬 실행됩니다.

프롬프트 엔지니어링 개선

초기에는 Agent의 backstory가 4~5줄 성격 묘사 수준이어서, Task description의 상세한 지시사항을 LLM이 일관되게 따르지 못하는 문제가 있었습니다. 이를 구조화된 프롬프트로 개선했습니다.
# ✗ Before: 추상적 성격 묘사 backstory = "당신은 친절한 기술 면접관입니다." # ✓ After: 구조화된 행동 규칙 + 판단 기준 backstory = """ ## 핵심 행동 원칙 1. 평가 결과 비공개: 점수를 멘트에 포함하지 않음 2. 답변 참조 연결: 후보자 키워드를 인용하여 연결 ## 액션 결정 규칙 - FOLLOW_UP: 꼬리질문 1회 미만 + 답변 모호 - NEXT_QUESTION: 꼬리질문 1회 완료 or 충분히 평가 가능 - END: 남은 메인 질문 0개 ## 출력 형식 반드시 JSON 구조로만 응답 """
이 패턴은 5개 에이전트 모두에 적용했으며, Agent backstory와 Task description이 핵심 제약을 상호 보강하는 구조로 설계했습니다. CrewAI는 backstory → Task description 순으로 컨텍스트를 구성하기 때문에, 양쪽에 핵심 규칙을 반복하면 LLM 준수율이 크게 향상됩니다.
03

pgvector RAG + 사용자별 지식 격리

pgvector RAG metadata 필터링
⚠ Problem
여러 사용자의 스터디 노트가 같은 벡터 저장소에 저장되어, similarity_search 시 다른 사용자의 노트가 함께 검색되는 데이터 격리 문제 발생
✅ Solution
임베딩 저장 시 metadata에 user_id를 포함하고, RAG 검색 Tool에서 user_id 필터링으로 해당 사용자의 노트만 similarity_search. CrewAI Tool 생성 시 user_id를 주입하여 에이전트가 자동으로 격리된 검색을 수행

임베딩 저장: metadata에 user_id 포함

# MD 노트 업로드 시 임베딩 저장 vectorstore.add_documents( documents=chunks, metadatas=[{ "user_id": user_id, "source": filename, "note_id": note_id, } for chunk in chunks] )

RAG 검색: user_id 필터링

class RAGSearchTool(BaseTool): def __init__(self, user_id: str): self.user_id = user_id # 팩토리에서 주입 def _run(self, query: str): results = vectorstore.similarity_search( query=query, k=5, filter={"user_id": self.user_id} # 격리! ) return results

CrewAI 에이전트에 Tool 주입

# Crew 생성 시 user_id를 Tool에 바인딩 rag_tool = RAGSearchTool(user_id=user_id) rag_analyst = create_analyst(user_id=user_id) task_c = Task( agent=rag_analyst, tools=[rag_tool], # user_id가 바인딩된 Tool async_execution=True, )
이 패턴으로 에이전트는 별도의 필터링 로직 없이 자연스럽게 해당 사용자의 노트만 검색합니다. 벡터 저장소를 물리적으로 분리하지 않아도 metadata 필터링만으로 안전한 논리적 격리를 달성했습니다.
